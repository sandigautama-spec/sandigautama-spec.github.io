ModernVet ETL Sync API

Push records from AWS S3 (or local files) into Odoo 18. Processing is asynchronous — trigger once, poll for result.

1 Authentication

Pass your API key in either header:

  • X-API-Key: <key>
  • Authorization: Bearer <key>

Keys are managed in Odoo under Settings → ETL API Keys (model: etl.api.key).

2 Trigger a Sync

POST a JSON body to /api/v1/etl/notifications:

  • data_type — what to sync
  • s3_location — S3 bucket/prefix or local path
  • local: true — read from filesystem

Returns {"status":"queued","log_id":42} immediately.

3 Poll for Status

GET /api/v1/etl/logs/{log_id} to check progress. States: queued → running → done / failed.

  • done — check created / updated / skipped / errors counters
  • failed — read error field for exception details

4 S3 Location Resolution

  1. Try s3_location as a direct object key
  2. If not found, list all objects under that prefix
  3. Download and process all matching files together

Multiple files are concatenated before parsing.

5 File Format

Files must be JSON array or JSONL (one object per line). Every record must have id_from_ext as the upsert key.

  • Encoding: UTF-8
  • Extension: .json or .jsonl
  • Max size: limited by available memory

6 Idempotency

All syncs are safe to re-run. Records are matched by id_from_ext:

  • Product / Sales / Invoice — upsert (create or update)
  • Deposit / Deposit Refund — create-only; existing id_from_ext is skipped
  • Loyalty History — create-only; existing id_from_ext is skipped
  • Posted/cancelled invoices — skipped on update attempt

7 AWS S3 Credentials

Configure in Odoo Settings → Technical → System Parameters:

  • pet_sync_etl.aws_region — default: ap-southeast-3
  • pet_sync_etl.aws_access_key_id
  • pet_sync_etl.aws_secret_access_key

Leave key/secret empty to use IAM role credentials.

8 Loyalty Journal Config

Required for loyalty journal entries. Configure per company in Settings → Accounting → Loyalty Configuration:

  • Invoice journal, payable account, revenue deduction account
  • Non-invoice journal, payable account, revenue deduction account

9 Stuck Jobs & Auto-Reset

If a job stays in running state for more than 2 hours, the next cron run auto-resets it to failed.

Cron runs every ~1 minute and processes one queued log at a time to prevent race conditions.

10 Warehouse id_from_ext

Sales, invoices, and products reference warehouses via warehouse_id (a string). The ETL looks up stock.warehouse by id_from_ext.

Set in Odoo: Inventory → Configuration → Warehouses — edit a warehouse and fill in the External ID field.

  • If not found: warehouse field is silently skipped (no error)
  • Analytic account is also resolved from the warehouse and applied to all lines

11 Payment Method Line id_from_ext

Invoice and refund payment_list[].payment_journal_code looks up account.payment.method.line by id_from_ext, scoped to the record's company and payment direction:

  • Invoice (data_type: "invoice") — looks up Incoming Payments method line. Set in Odoo: Accounting → Configuration → Journals → open a journal → Incoming Payments tab → fill in External ID.
  • Refund (data_type: "refund") — looks up Outgoing Payments method line. Set in Odoo: Accounting → Configuration → Journals → open a journal → Outgoing Payments tab → fill in External ID.
  • Deposit (data_type: "deposit") — looks up Incoming Payments method line (same setup as Invoice).
  • Deposit Refund (data_type: "deposit_refund") — looks up Outgoing Payments method line (same setup as Refund).
  • If not found: that payment entry is skipped with a warning in the log — credit note is posted but unpaid

12 Special Products & Loyalty Program

These Odoo records must exist (created by the module's data files on install):

  • pet_sync_etl.product_global_discount — product used for the global discount line on sales/invoices
  • pet_sync_etl.product_loyalty_redemption — product used for the loyalty redemption line on invoices
  • pet_sync_etl.loyalty_program_modernvet — loyalty program used for card & history sync

If any of these are missing, the related feature is silently skipped (no error, just a warning in the server log).

Invoice
data_type: "invoice" or "invoices"  ·  Target model: account.move  ·  Upsert by id_from_ext
ℹ️
Auto-post flow: When state=posted, the ETL will:
  1. Add voucher line if voucher_amount is set — negative line using voucher_product_id; label = voucher_code
  2. Add loyalty redemption line if point_redeem_idr > 0
  3. Add global discount line if global_discount < 0
  4. Post the invoice
  5. Create loyalty card history + journal entries
  6. Register payments from payment_list:
    • If entry has payment_journal_code → creates account.payment (inbound); actual amount = payment_list.amount − loyalty redemption line − global discount line (voucher is already pre-deducted in amount)
    • If entry has payment_id_from_ext → applies an existing deposit credit via a transfer journal entry (no new payment created)

Refund / Credit Note
data_type: "refund" or "refunds"  ·  Target model: account.move (out_refund)  ·  Create-only by id_from_ext
⚠️
Create-only: Records where id_from_ext already exists are silently skipped — no update is performed. reversed_entry_id must reference an invoice that exists and is fully paid (in_payment or paid), otherwise the record is skipped.
ℹ️
Same JSON format as invoice. move_type is not required — data_type: "refund" always creates out_refund. When state=posted, the ETL will:
  1. Post the credit note
  2. Register outgoing payments from payment_list (method line matched by id_from_ext + payment_type=outbound)
  3. Deduct earned loyalty points (point_earning) — JE: Debit Payable, Credit Revenue Deduction
  4. Restore redeemed loyalty points (point_redeem) — JE: Debit Revenue Deduction, Credit Payable

Step 6 Refund / Credit Note (REFUND-2026-001)
📋
Creates a credit note reversing INV-2026-001. Requires the original invoice to be fully paid (in_payment or paid).
The ETL adds a Loyalty Redemption line (−50,000) automatically from point_redeem_idr.
Payment deduction: actual_payment = 1,035,000 − 50,000 (redeem) = 985,000.
Loyalty earn deduction JE (−2,000 pts) and redeem restoration JE (+5,000 pts) are created after posting.
Note: the credit note is a standalone document — it is not reconciled against the original invoice (which is already fully paid).
⚠️
Outgoing payment method setup required: payment_journal_code must match id_from_ext on an Outgoing account.payment.method.line. Go to Accounting → Configuration → Journals → Outgoing Payments tab → External ID column and fill in the value.

Step 7 Deposit → Invoice with Deposit Applied
💰
Two separate syncs. First, create the deposit payment (data_type: "deposit"). Then, create the invoice (data_type: "invoice") with a payment_list entry that references the deposit via payment_id_from_ext.

The ETL creates a transfer journal entry: Debit Customer Deposit Account → Credit Receivable, then reconciles the deposit credit line with the transfer debit line. The invoice residual drops by the applied amount.

Amounts: Deposit = 500,000. Invoice = 300,000. Applied = 300,000 → residual = 0. Remaining deposit balance = 200,000.
